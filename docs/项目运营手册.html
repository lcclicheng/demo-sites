01项目定位与价值主张

我们是谁：一个人运营的建站服务（独立开发者 Ethan Li），面向英国小商家（餐厅、咖啡馆、沙龙、瑜伽馆、律所、甜品店、维修、精品酒店、花店、摄影、宠物美容、书店等）。

我们卖什么：一个高端、可上线、零托管成本的静态站点，由我们替客户填内容、一键生成、自动部署。客户不懂开发，全程我们代劳。

客户痛点

小商家预算有限、没技术团队、现有建站方案贵且难维护（ freelancer £1k–3k 一次性 + 维护 £300–1.2k/年，或笨重 CMS 要自管）。

我们的解法

Vite+React+Tailwind 模板引擎 + GitHub Pages 零服务器成本；进攻型两档定价；首月代改、之后走年费呵护，把维护负担压到极低。

差异化

从 20 个旧独立站提炼的 editorial 编辑型视觉语言（反网红、亮暗主题自适应）；统一设计系统而非套模板感；专业交付包提升信任。

03站点生成与部署步骤

新客户接入与上线是标准化 SOP（客户不懂开发，全程我们代填）：

  1. 启动接入工具：node onboard.mjs → 浏览器 http://localhost:4321/。
  2. 选模板克隆：下拉选同行业演示站，自动载入该模板全字段。
  3. 业务类型切换：Ltd / Sole Trader 一键切换（Sole Trader 清空 registeredAddress）。
  4. 填表：递归渲染全部字段（名称、电话、地址、菜单、评价、关于、营业时间…），数组字段可增删，不碰 JSON 语法。
  5. 上传实景图：自动写入 assets/<slug>/，无需手动放文件夹。
  6. 生成并构建：点「生成」→ 写出 examples/<slug>.json + 单站 generate.mjs 自动构建（30–60s，失败重试一次）。
  7. 全量构建 + 部署：bash build-clean.sh → git add -u && git commit && git push origin main。
  8. 交付客户：发线上地址 + 交付标准化包（见 §09）。
交付后维护（§5.9）：客户经微信/邮件把变更发我们 → 改 examples/<slug>.json（或覆盖 assets/<slug>/ 同名图）→ bash build-clean.sh → commit → push。图片带 ?v=<md5> 内容哈希，覆盖即刷新。紧急改动可走单站快速构建：node generate.mjs "./examples/<slug>.json"（仅本地验证，上线仍需 push）。

质量闸门（部署前 fail-fast）

validate-sites.mjs 逐站校验：① JSON 合法 ② 必填 name/slug/template 齐全 ③ 图片引用真实存在。任一不满足 → exit 1 → 整 job 失败，残缺内容绝不上线。孤儿站点（有 template+slug 但未进 PROJS）默认软阻断 exit 1。

部署认证（已根治）

走 SSH（443 通道）：~/.ssh/id_ed25519_github + config: github.com → ssh.github.com:443。无需 PAT、无 workflow scope 坑。ssh -T git@github.com 返回 Hi lcclicheng! 即通。

08交付与售后

交付标准化包（docs/delivery-handover.md，纯英文、面向 UK 客户）把一次交付变成专业 onboarding。含 7 模块：

  1. Welcome Email：告知上线 + 首月无限改版 + 维护指南 + 支持条款。
  2. Compliance Sign-off：隐私政策/GDPR/ECCTA 注册地址/公开信息警示/图片版权/域名 — 客户签字回收。
  3. Maintenance Guide（一页纸）：如何提修改、续费域名、联系我们——无后台、无登录。
  4. Custom Domain Handover：客户自管 registrar/DNS，我们仅协助。
  5. Support Terms：响应时间、首月无限改、年费免费小改动额度、计费改版单价、Annual Care £390/年。
  6. Invoice & Payment：发票模板（£ 金额、支付渠道）。
  7. Tier C Full Ownership Handover：源码+数据+Decap（指向客户自己仓库）移交，运维归客户。

12风险与边界

  • 隐私边界：GitHub Pages 站点完全公开，电话/地址对外可见；交付前提醒客户。Tier C 客户自管仓库后由客户负责。商家清单与 business/ 含内部口径，gitignored 不进公开仓库。
  • 法律边界：不替客户选/担保注册地址合法性（ECCTA 2024「适当地址」）；隐私政策模板为占位，客户须补真实条款。AI 只做模板注入。
  • 密钥安全：SSH 私钥绝不提交 git；定期轮换；GITHUB_TOKEN 权限最小化（contents:read + pages:write + id-token:write）。
  • 构建坑：预览服务器锁 dist/ 致 EBUSY 假成功（关服务再重建）；图片目录缺失静默跳过（校验闸门拦截）；单站 build 只本地、上线仍需 push。
  • GitHub Pages 限制：仓库 <1GB、月带宽 ~100GB；图片 WebP/压 <300KB；构建超阈值建议 fail 部署。
  • 发送卫生：真实邮箱仅存私有清单；demo 占位邮箱不可直发；分散 5 家/天保护发件账号。
  • 商业系统缺口已补：Business Layer（business/）已建立 Lead Pipeline / CRM 状态机 / KPI / 客户成功 / 增长闭环；下一步靠真实发送数据回填 KPI 基线，并用 Customer Success 节奏拉高续费。
